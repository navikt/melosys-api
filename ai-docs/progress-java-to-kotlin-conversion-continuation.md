## Technical Patterns Established

### MockK Patterns

- `@MockK` for service dependencies
- `every { service.method(any()) } returns value` for method mocking
- `every { service.method(any()) } just Runs` for void methods
- `answers { firstArg<Type>() }` for repository save methods

### Kotest Assertion Patterns

- `shouldBe` for equality assertions
- `shouldContainExactly` for list comparisons
- `shouldContainExactlyInAnyOrder` for order-independent list comparisons
- `shouldHaveSize` for collection size assertions
- `shouldBeEmpty()` for empty collection assertions

### Domain Class Access Patterns

- Direct property access instead of Java getter/setter methods
- `list = list + element` for immutable list modifications
- `?.` for null-safe property access
- `toMutableList()` for creating mutable copies of immutable lists
